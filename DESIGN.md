# dotslops

Dotslops is a specification and tooling to notate LLM generated code in a monorepo that is mostly human written.

## Why

- I believe (most) LLM generated code is sub-par compared to a good programmer.
- However - also there are cases where the sub-par code is "good enough":
  - Scripts that does not serve a critical function
  - Utility code that serves a single purpose and is behind a strict interface
  - Experimental interfaces, internal UI styling etc.
- However - in a monorepo setup usually those "slops" are mixed with "good code" and it causes mental burden and confusion for developers.
- By notating "slops" and using supporting tooling we can allow the developer to distinguish them at a glance.
- It also allows us to maintain the prompts that generated & edited the code.

### Clarification

- All LLM generated code is not "slops". If it went through (multiple) human reviews and well maintained - it can be considered "good code". Being a "slop" is less about how the code is written and more about how much human attention it deserves.

## Tooling

- `dotslops` CLI tool that can be used to mark files as slops, and some utilities to browse & manage the `.slops` directory.
- A CLAUDE.md hook that marks all files it edited as "slops".
- A VSCode extension that marks the "slop" files, and can show the prompt(s) that generated a "slop".

## Specification

### File header

All "slops" should have a header within the first lines of the file with the following format:

```
@dotslops [PROMPTID] [tag...]
```

It can optionally be preceded by `//`, `#`, optionally followed by whitespace.

#### Tags

Tags are optional and can be used to further classify "slops". Here are some tags that are conventionally used:

- `heavy`: Means the file is created or heavily edited by the prompt.
- `light`: Means only a small portion of the file is edited by the prompt.

### .slops

The `.slops` is a directory that contains the "prompts" that generated the "slops". There is a many-to-many relationship between "slops" and "prompts". A "slop" can be generated by zero to many prompts, and a prompt can generate multiple "slops".

Slops are named by their `PROMPTID` and are plain text files.

## FAQ

### Specification

- Why not .gitattributes
  - It has a "more specific attribute" wins logic, which means managing slops with other .gitattributes functionality is non-trivial.
  - Having the prompt id in the files make handling moving files easier.

- Why not a comment with a specific format instead of prompts under .slops/
  - Single prompt can modify multiple files.
  - A file can be modified by multiple prompts and it would be noisy to have all that information inth efile.
